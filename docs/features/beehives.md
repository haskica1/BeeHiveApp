# Feature: Beehives

## Overview

A beehive belongs to an **Apiary**. Each beehive has a type, material, a unique GUID identifier, and a QR code (Base64 PNG). Beehives are the central entity — inspections, diets, and (optionally) todos attach to them.

## Domain Rules

- A beehive must belong to exactly one apiary; apiary cannot be changed after creation
- `UniqueId` (Guid) is generated on creation and never changes — used for QR code scanning
- QR code encodes the `UniqueId` as a Base64 PNG, generated by `QrCodeService` at creation time
- Deleting a beehive cascades to inspections, diets, and beehive-scoped todos

## Enums

```
BeehiveType:     Langstroth | DadantBlatt | Warré | TopBar | Other
BeehiveMaterial: Wood | Plastic | Polystyrene
```

## API

- `GET /beehives/by-apiary/{apiaryId}` — list all beehives in an apiary
- `GET /beehives/{id}` — detail including recent inspections list
- `POST /beehives` — creates beehive, auto-generates `uniqueId` and `qrCode`
- `PUT /beehives/{id}` — update name, type, material only
- `DELETE /beehives/{id}` — cascade delete

## UI Rules

- QR code displayed on beehive detail page as an `<img src="data:image/png;base64,..."/>`
- Beehive list shows type + material badges
- Beehive detail page embeds: inspection list, diet section, todo section
- "Add Inspection", "Add Diet", "Add Todo" buttons contextually pass `beehiveId` to forms

## Edge Cases

- A beehive with active inspections or diets can still be deleted (cascade handles cleanup)
- `qrCode` field can be large (Base64 PNG) — do not include in list responses, only in detail
- If QR generation fails, the beehive should still be created (QR can be regenerated)

## Implementation Notes

- `BeehiveDetailDto` includes `InspectionDto[]` — ordered newest first by service layer
- `QrCodeService` uses `QRCoder` library, outputs 300×300 PNG
- `uniqueId` is exposed in the DTO for use in QR scanning flows
